# Internationalization (i18n + RTL)

`@sagtech-infra/ui` is **English / LTR by default**. Locale-aware date formatting and
right-to-left layout are opt-in via `LocaleProvider` (or `SagtechUIProvider`).
Consumers that do nothing keep today's behavior exactly (`en-US`, `ltr`).

---

## 1. `LocaleProvider` + `useLocale()`

`LocaleProvider` publishes a `{ locale, dir }` context. It is **controlled-only** —
no persistence, no `matchMedia`, no DOM mutation — so it is SSR-deterministic and
needs no anti-FOUC script (unlike `ThemeProvider`).

```tsx
import { LocaleProvider } from '@sagtech-infra/ui';

<LocaleProvider locale="de-DE" dir="ltr">
  <YourApp />
</LocaleProvider>
```

### Props

| Prop | Type | Default | Description |
|---|---|---|---|
| `locale` | `string` (BCP-47) | `"en-US"` | Locale for date/number formatting. |
| `dir` | `"ltr" \| "rtl"` | `"ltr"` | Text direction; components apply it to their root. |

### `useLocale()`

```tsx
import { useLocale } from '@sagtech-infra/ui';

const { locale, dir } = useLocale(); // { locale: 'en-US', dir: 'ltr' } with no provider
```

With **no provider**, `useLocale()` returns the defaults `{ locale: 'en-US', dir: 'ltr' }`,
which is why unwrapped apps are unaffected.

### Via `SagtechUIProvider`

If you already use `SagtechUIProvider`, pass `locale`/`dir` there to avoid a nested
provider (mirrors how `theme`/`defaultTheme` compose):

```tsx
<SagtechUIProvider theme="dark" locale="ar-EG" dir="rtl">
  <YourApp />
</SagtechUIProvider>
```

`SagtechUIProvider` wraps children in a `LocaleProvider` only when `locale` or `dir`
is provided; otherwise it skips the wrapper entirely.

## 2. Locale-aware dates

`DatePicker`, `DateRangePicker`, and `GanttTimeline` localize their weekday and month
labels through `Intl.DateTimeFormat`. Each resolves its locale as:

**`locale` prop → `LocaleProvider` context → `"en-US"`**

```tsx
// Provider-driven
<LocaleProvider locale="de-DE">
  <DatePicker value={date} onChange={setDate} />
</LocaleProvider>

// Per-instance override (wins over the provider)
<DatePicker value={date} onChange={setDate} locale="fr-FR" />
```

- Weekday headers come from `getWeekdayLabels(locale)` and the month label from
  `Intl.DateTimeFormat(locale, { month: 'long', year: 'numeric' })`.
- **The calendar grid stays Monday-first** in every locale — only the label glyphs
  change, not the column order.
- `GanttTimeline` tick labels (weekday/month) also follow the resolved locale.

> **Note:** `GanttTimeline` previously formatted tick labels with the raw browser
> locale; from v1.5 it defaults to the resolved locale (`en-US` when unwrapped).
> This affects tick **labels** only — never the timeline geometry.

## 3. RTL support

When `dir` is `"rtl"`, RTL-aware components set `dir="rtl"` on their root (and, for
portalled overlays, on the portal-subtree root so it survives the portal boundary).
Layout flips automatically because these components use **Tailwind v4 logical
properties** (`ps/pe`, `ms/me`, `start/end`, `border-s/e`) instead of physical
`left/right` ones; a few CSS-triangle arrows and slide directions are mirrored with
`rtl:` variants / `dir`-aware offsets.

Enable RTL by providing `dir="rtl"` (via `LocaleProvider` or `SagtechUIProvider`), or
simply by setting `dir="rtl"` on an ancestor element for the leaf form controls.

### RTL-audited components

- **Form controls:** `Input` (floating label), `SelectInput` (chevron).
- **In-tree overlays:** `Tooltip`, `Popover` (incl. arrows), `Drawer` (edge + slide
  direction), `Sheet` (edge + slide direction), `DateRangePicker`.
- **Portalled overlays:** `Modal`, `Toaster`, `BottomSheet`, `DatePicker` calendar —
  `dir` is applied on the element rendered inside the portal.

```tsx
<SagtechUIProvider dir="rtl" locale="ar-EG">
  <App /> {/* Drawer slides from the inline-end edge, Toaster anchors inline-end, etc. */}
</SagtechUIProvider>
```

## 4. Known limitations

These are intentional v1.5 scope boundaries (tracked for a future release):

- **`GanttTimeline` axis is not mirrored in RTL.** The timeline geometry uses absolute
  pixel math (`left:` offsets), so the axis stays left-to-right; only tick **labels**
  localize.
- **Canvas charts are not locale/RTL/theme-reactive.** The 10 chart components draw
  with JavaScript color values on `<canvas>`, so their colors and text direction do
  not follow the active theme or `dir`. Theming the canvas layer is a separate effort.
- **`NotificationCenter` dropdown alignment** uses physical `left/right` anchoring and
  was not part of the RTL audit; its content localizes, but its open-side does not flip.
- The internal 4px directional fade offset on `Popover`/`Tooltip` is not mirrored in
  RTL (imperceptible on a fade).
